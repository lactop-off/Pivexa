## 3. 分析フレーム契約（最重要）

全手法が実装する抽象基底。フロント・API・worker はこの契約のみに依存し、個別手法を知らない。

### 3.1 抽象基底クラス

```python
# api/analysis/base.py
from abc import ABC, abstractmethod
import pandas as pd

class AnalysisMethod(ABC):
    name: str           # 例: "linear_regression"（一意）
    display_name: str   # 例: "重回帰分析"
    needs_target: bool  # 目的変数が必要か
    target_kind: str | None   # "numeric" | "binary" | "categorical" | None
    min_rows: int       # 最低必要行数

    @abstractmethod
    def config_schema(self, dataset: "DatasetProfile") -> "ConfigSchema":
        """設定画面に必要な入力定義を返す（選択可能な変数候補を含む）"""

    @abstractmethod
    def validate(self, config: "AnalysisConfig", dataset: "DatasetProfile") -> "ValidationResult":
        """設定とデータの整合性を検証。エラー/警告のリストを返す"""

    @abstractmethod
    def run(self, config: "AnalysisConfig", df: pd.DataFrame) -> "AnalysisResult":
        """解析を実行し共通形式の結果を返す"""

    @abstractmethod
    def interpret(self, result: "AnalysisResult") -> "Interpretation":
        """結果からテンプレート解釈文を生成する"""
```

### 3.2 レジストリ

```python
# api/analysis/registry.py
_REGISTRY: dict[str, AnalysisMethod] = {}

def register(method: AnalysisMethod) -> None:
    _REGISTRY[method.name] = method

def get_method(name: str) -> AnalysisMethod:
    if name not in _REGISTRY:
        raise UnknownMethodError(name)
    return _REGISTRY[name]

def list_methods() -> list[AnalysisMethod]:
    return list(_REGISTRY.values())
```

各手法モジュールは末尾で `register(...)` を呼ぶ。`methods/__init__.py` で全手法を import して登録を確定させる。**新手法追加 = methods/ にファイル追加 + import 1行**。

### 3.3 共通データ構造（Pydantic）

```python
# api/analysis/schema.py

class ConfigField(BaseModel):
    key: str                  # "target" | "explanatory" | "group" など
    label: str
    kind: str                 # "single_select" | "multi_select" | "option"
    required: bool
    candidates: list[str]     # 選択可能な列名（手法が型でフィルタ済み）

class ConfigSchema(BaseModel):
    method: str
    fields: list[ConfigField]

class AnalysisConfig(BaseModel):
    method: str
    target: str | None = None
    explanatory: list[str] = []
    options: dict[str, Any] = {}   # 手法固有（group列、有意水準など）

class Issue(BaseModel):
    level: str                # "error" | "warning"
    message: str              # 日本語

class ValidationResult(BaseModel):
    ok: bool
    issues: list[Issue] = []

# --- 結果の共通スキーマ ---
class Metric(BaseModel):
    key: str
    label: str
    value: float | int | str
    significant: bool | None = None   # 有意フラグ（見える化用）

class CoefficientRow(BaseModel):       # 回帰系で使用
    variable: str
    coef: float
    std_err: float | None = None
    stat: float | None = None          # t値/z値
    p_value: float | None = None
    ci_low: float | None = None
    ci_high: float | None = None
    extra: dict[str, float] = {}       # オッズ比/VIF など

class ChartRef(BaseModel):
    kind: str                 # "histogram" | "scatter" | "heatmap" | "bar" | "roc" | "confusion"
    label: str
    path: str                 # reports ボリューム内の PNG パス

class AnalysisResult(BaseModel):
    method: str
    summary_metrics: list[Metric] = []        # モデル全体指標
    coefficients: list[CoefficientRow] = []    # 変数別（任意）
    tables: dict[str, Any] = {}                # クロス集計表など手法固有
    charts: list[ChartRef] = []
    sample_size: int
    warnings: list[str] = []

class InterpretSentence(BaseModel):
    level: str                # "info" | "highlight" | "caution"
    text: str

class Interpretation(BaseModel):
    sentences: list[InterpretSentence]
```

この共通スキーマにより、フロントは `summary_metrics` / `coefficients` / `charts` / `interpret` を**手法非依存に描画**できる。

## 4. 解釈エンジン（テンプレート方式）

### 4.1 方針
- 指標値を閾値判定して定型文を選び、変数名・数値を差し込む
- 共通の注意喚起を末尾に付与
- 外部 LLM 非依存（オンプレ完結）

### 4.2 共通テンプレート部品

```python
# api/analysis/interpret/common.py
P_SIGNIFICANT = 0.05

def significance_sentence(var: str, p: float) -> InterpretSentence:
    if p < 0.01:
        return InterpretSentence(level="highlight",
            text=f"「{var}」は統計的に強く有意です（p={p:.3f}）。偶然とは考えにくい関係があります。")
    if p < P_SIGNIFICANT:
        return InterpretSentence(level="highlight",
            text=f"「{var}」は統計的に有意です（p={p:.3f}）。")
    return InterpretSentence(level="info",
        text=f"「{var}」は統計的に有意とは言えません（p={p:.3f}）。")

def small_sample_caution(n: int, threshold: int = 30) -> InterpretSentence | None:
    if n < threshold:
        return InterpretSentence(level="caution",
            text=f"サンプル数が{n}件と少なめです。結果は参考程度にとどめてください。")
    return None

CAUSALITY_NOTE = InterpretSentence(level="caution",
    text="統計的な関係は必ずしも因果関係を意味しません。結果の解釈にはご注意ください。")
```

各手法の `interpret()` はこれら部品＋手法固有文を組み合わせる（個別手法編で定義）。

## 5. 共通基盤モジュール詳細

### 5.1 dataset（取り込み・プロファイリング）

| 機能 | 処理 |
|---|---|
| アップロード | 形式判定（拡張子＋中身）、サイズ検証、文字コード/区切り判定、空・壊れ検出、`uploads`へ保存、`datasets`/`dataset_columns`登録 |
| プロファイリング | 列ごとに型推定（numeric/datetime/categorical）、欠損数、基本統計、ユニーク率、重複行数を算出し JSON 保存 |
| 推奨値生成 | 1章の閾値に基づき列ごとの推奨アクションを付与 |

型推定ロジック（優先順）：
1. 数値変換成功率 ≥ 95% → `numeric`
2. 日付変換成功率 ≥ 95% → `datetime`
3. それ以外 → `categorical`

### 5.2 preprocess（前処理）

設定（JSON）を保存し、解析実行時に DataFrame へ適用する。適用は非破壊（元ファイルは保持、派生 DataFrame を生成）。

```json
{
  "operations": [
    {"type": "impute", "column": "age", "method": "median"},
    {"type": "impute", "column": "region", "method": "mode"},
    {"type": "drop_column", "column": "user_id"},
    {"type": "cast", "column": "price", "to": "numeric"}
  ]
}
```

カテゴリ変数のダミー変数化は**解析実行時に自動**で行う（pandas `get_dummies`、基準カテゴリを1つ除外）。

### 5.3 job（ジョブ管理）

- `POST /jobs`：`validate()` を同期実行→OKならジョブ登録。同期閾値内なら即時実行、そうでなければ Celery enqueue
- Celery タスク：前処理適用→`get_method(name).run()`→グラフ生成→`analysis_results` 保存→状態更新
- 失敗時は状態 `error` とメッセージを保存（日本語の利用者向け文言）

### 5.4 report（レポート出力）

- `AnalysisResult` + `Interpretation` を HTML テンプレートに流し込み、WeasyPrint で PDF 化
- グラフ PNG は埋め込み
- 画像（PNG）単体エクスポートも提供

## 6. DB 物理スキーマ

PostgreSQL。可変構造は `JSONB`。タイムスタンプは `timestamptz`。

```sql
CREATE TABLE users (
  id            BIGSERIAL PRIMARY KEY,
  username      VARCHAR(100) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  role          VARCHAR(20)  NOT NULL DEFAULT 'admin',
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE TABLE datasets (
  id            BIGSERIAL PRIMARY KEY,
  name          VARCHAR(255) NOT NULL,
  original_name VARCHAR(255) NOT NULL,
  format        VARCHAR(10)  NOT NULL,          -- 'csv' | 'excel'
  row_count     INTEGER,
  col_count     INTEGER,
  file_path     VARCHAR(500) NOT NULL,
  created_by    BIGINT REFERENCES users(id),
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE TABLE dataset_columns (
  id            BIGSERIAL PRIMARY KEY,
  dataset_id    BIGINT NOT NULL REFERENCES datasets(id) ON DELETE CASCADE,
  name          VARCHAR(255) NOT NULL,
  inferred_type VARCHAR(20)  NOT NULL,          -- 'numeric'|'datetime'|'categorical'
  missing_count INTEGER NOT NULL DEFAULT 0,
  summary       JSONB,                          -- 統計サマリー・推奨アクション
  UNIQUE (dataset_id, name)
);

CREATE TABLE preprocessings (
  id            BIGSERIAL PRIMARY KEY,
  dataset_id    BIGINT NOT NULL REFERENCES datasets(id) ON DELETE CASCADE,
  operations    JSONB NOT NULL,
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE TABLE analysis_jobs (
  id            BIGSERIAL PRIMARY KEY,
  dataset_id    BIGINT NOT NULL REFERENCES datasets(id) ON DELETE CASCADE,
  method        VARCHAR(50)  NOT NULL,
  config        JSONB NOT NULL,
  status        VARCHAR(20)  NOT NULL DEFAULT 'queued',  -- queued|running|done|error
  error_message TEXT,
  created_by    BIGINT REFERENCES users(id),
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now(),
  finished_at   TIMESTAMPTZ
);
CREATE INDEX idx_jobs_status ON analysis_jobs(status);
CREATE INDEX idx_jobs_dataset ON analysis_jobs(dataset_id);

CREATE TABLE analysis_results (
  id            BIGSERIAL PRIMARY KEY,
  job_id        BIGINT NOT NULL REFERENCES analysis_jobs(id) ON DELETE CASCADE,
  result        JSONB NOT NULL,                 -- AnalysisResult をシリアライズ
  interpretation JSONB NOT NULL,                -- Interpretation
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX idx_results_job ON analysis_results(job_id);

CREATE TABLE reports (
  id            BIGSERIAL PRIMARY KEY,
  result_id     BIGINT NOT NULL REFERENCES analysis_results(id) ON DELETE CASCADE,
  format        VARCHAR(10) NOT NULL,           -- 'pdf' | 'png'
  file_path     VARCHAR(500) NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE audit_logs (
  id            BIGSERIAL PRIMARY KEY,
  user_id       BIGINT REFERENCES users(id),
  action        VARCHAR(100) NOT NULL,
  target        VARCHAR(255),
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 7. API I/O 仕様

共通エラー形式：

```json
{ "error": { "code": "VALIDATION_ERROR", "message": "日本語の説明", "details": [] } }
```

主要エンドポイントの I/O（抜粋）：

### POST `/auth/login`
- Req: `{ "username": "...", "password": "..." }`
- Res: `200` HttpOnly Cookie に JWT 設定、`{ "user": { "id", "username", "role" } }`

### POST `/datasets`（multipart）
- Req: file（CSV/Excel）, name（任意）
- Res: `201 { "id", "name", "format", "row_count", "col_count" }`
- Err: `400`（形式不正・サイズ超過・空/壊れ）

### GET `/datasets/{id}/profile`
- Res:
```json
{
  "dataset_id": 1,
  "columns": [
    {"name": "age", "type": "numeric", "missing": 12,
     "summary": {"mean": 34.2, "min": 18, "max": 80, "p25": 27, "p50": 33, "p75": 41},
     "recommendation": {"action": "impute", "method": "median", "reason": "数値列に欠損があります"}},
    {"name": "user_id", "type": "categorical", "missing": 0,
     "summary": {"unique_ratio": 0.99},
     "recommendation": {"action": "drop_column", "reason": "ID的な列のため説明変数に不適"}}
  ]
}
```

### GET `/methods`
- Res: 各手法の `ConfigSchema`（`needs_target`, `target_kind`, `fields`）一覧

### POST `/jobs`
- Req: `{ "dataset_id": 1, "method": "linear_regression", "config": { "target": "sales", "explanatory": ["ad_cost","temp"], "options": {} } }`
- Res（非同期）: `202 { "job_id": 10, "status": "queued" }`
- Res（同期実行時）: `200 { "job_id": 10, "status": "done", "result_id": 5 }`
- Err: `422`（validate のエラー、`issues` を返却）

### GET `/jobs/{id}`
- Res: `{ "job_id":10, "status":"running", "result_id": null }` / 完了時 `result_id` 設定

### GET `/results/{id}`
- Res: `AnalysisResult` + `Interpretation`（共通スキーマのシリアライズ）

### POST `/results/{id}/report`
- Req: `{ "format": "pdf" }`
- Res: `201 { "report_id": 7, "format": "pdf" }`

## 9. 個別手法編への引き継ぎ

個別手法編では、各手法について以下を定義する。

1. `config_schema`：必要な入力（target/explanatory/group/options）
2. `validate`：手法固有の検証（型・群数・件数）
3. `run`：使用ライブラリ関数、算出する `summary_metrics` / `coefficients` / `tables` / `charts`
4. `interpret`：手法固有の解釈テンプレート文
