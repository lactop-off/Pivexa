## 1. システム構成

### 1.1 全体構成（Docker Compose）

MVP は単一ホスト上の Docker Compose で構成する。テナント分離は行わない（単一組織前提）。

```
┌─────────────────────────────────────────────────────────────┐
│                        ホスト（顧客オンプレ環境）                  │
│                                                                 │
│   ┌──────────┐    ┌──────────────┐    ┌──────────────────┐    │
│   │  Nginx    │───▶│  frontend     │    │  api (FastAPI)    │    │
│   │ (reverse  │    │ (Next.js)     │    │  - 認証/ユーザー    │    │
│   │  proxy)   │───────────────────────▶│  - データ/前処理    │    │
│   └──────────┘                         │  - 分析ジョブ受付    │    │
│        │                               │  - 結果/レポート     │    │
│        │ HTTPS                         └─────────┬─────────┘    │
│   （外部からの接続）                              │              │
│                              ┌──────────────────┼────────┐     │
│                              ▼                  ▼        ▼     │
│                     ┌──────────────┐  ┌──────────┐ ┌────────┐ │
│                     │ worker        │  │ PostgreSQL│ │ Redis  │ │
│                     │ (Celery)      │  │           │ │(broker)│ │
│                     │ 解析エンジン    │  └──────────┘ └────────┘ │
│                     │ pandas/        │        ▲                 │
│                     │ statsmodels/   │        │                 │
│                     │ scikit-learn   │  ┌──────────────┐        │
│                     └───────┬────────┘  │ volume        │        │
│                             └──────────▶│ - DB data     │        │
│                                         │ - upload files│        │
│                                         │ - reports     │        │
│                                         └──────────────┘        │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 コンテナ一覧

| サービス | 技術 | 役割 |
|---|---|---|
| `nginx` | Nginx | リバースプロキシ、HTTPS終端、静的配信 |
| `frontend` | Next.js (React) | UI（SSR/CSR）。API を呼び出す |
| `api` | FastAPI (Python) | REST API。認証・データ管理・ジョブ受付・結果取得 |
| `worker` | Celery (Python) | 解析処理の非同期実行（重い計算を担当） |
| `db` | PostgreSQL | ユーザー・データセットメタ・ジョブ・結果メタ |
| `redis` | Redis | Celery のブローカー兼結果バックエンド |

### 1.3 永続化ボリューム

| ボリューム | 内容 |
|---|---|
| `db-data` | PostgreSQL データ |
| `uploads` | アップロードされた CSV/Excel（解析対象データ本体） |
| `reports` | 生成済み PDF/画像レポート |

### 1.4 同期/非同期の方針

- 軽量な処理（プロファイリング、小サイズの記述統計）は API 内で同期実行も許容
- 回帰・分散分析など重い処理、または大きいデータは **Celery ジョブ**として非同期実行
- ジョブ状態は DB と Redis で管理し、フロントはポーリングで進捗・完了を取得

## 2. モジュール構成

### 2.1 設計の核：共通分析フレーム＋個別手法プラグイン

要件定義「共通設計方針」に従い、**「設定 → 実行 → 結果 → 解釈」を共通フレーム化**し、各手法をプラグインとして差し替え可能にする。

```
api/
├── core/                  # 共通基盤
│   ├── auth/              # 認証・ユーザー
│   ├── dataset/           # アップロード・プロファイリング・前処理
│   ├── job/               # ジョブ受付・状態管理
│   └── report/            # レポート生成（PDF/画像）
├── analysis/
│   ├── base.py            # 分析手法の抽象基底（AnalysisMethod）
│   ├── registry.py        # 手法レジストリ（手法名→実装の解決）
│   ├── schema.py          # 設定/結果の共通スキーマ
│   ├── interpret/         # テンプレート解釈エンジン
│   └── methods/           # 個別手法プラグイン
│       ├── descriptive.py
│       ├── correlation.py
│       ├── crosstab_chi2.py
│       ├── ttest.py
│       ├── anova.py
│       ├── linear_regression.py
│       └── logistic_regression.py
└── main.py
```

### 2.2 分析手法の共通インターフェース（概念）

各手法は同一の契約を実装する。詳細なシグネチャは詳細設計で定義。

| メソッド/属性 | 役割 |
|---|---|
| `name` / `display_name` | 手法の識別子・表示名 |
| `config_schema()` | 設定画面が必要とする入力項目（目的変数/説明変数の要否など）を返す |
| `validate(config, dataset)` | 設定とデータの整合性チェック（変数の型・件数など） |
| `run(config, dataframe)` | 解析を実行し、共通形式の結果オブジェクトを返す |
| `interpret(result)` | 結果からテンプレート解釈文を生成 |

この契約により、フロントは手法に依存せず「設定 → 実行 → 結果 → 解釈」を共通描画できる。新手法の追加は `methods/` への1ファイル追加＋レジストリ登録で完結する。

### 2.3 解釈エンジン（テンプレート方式）

- 指標（p値・係数・R²・サンプル数など）の閾値に応じて定型文を選択し、変数名を差し込む
- 共通の注意喚起（相関と因果の違い、サンプル数不足の警告）を付与
- AI 生成は行わない（外部 LLM 依存なし＝オンプレで完結）

## 4. データベース概要

論理エンティティのみ示す（カラム詳細・型・制約は詳細設計）。

| エンティティ | 役割 | 主な属性（概要） |
|---|---|---|
| `users` | ユーザー | id, ユーザー名, パスワードハッシュ, ロール（admin）, 作成日時 |
| `datasets` | アップロードデータのメタ | id, 名称, 元ファイル名, 形式(csv/excel), 行数, 列数, ファイルパス, 作成者, 作成日時 |
| `dataset_columns` | 列メタ（プロファイル結果） | id, dataset_id, 列名, 推定型, 欠損数, 統計サマリー(JSON) |
| `preprocessings` | 前処理設定 | id, dataset_id, 操作内容(JSON: 補完/削除/型変換/列除外) |
| `analysis_jobs` | 解析ジョブ | id, dataset_id, 手法, 設定(JSON), 状態(queued/running/done/error), 作成者, 作成/完了日時 |
| `analysis_results` | 解析結果 | id, job_id, 指標(JSON), グラフ参照, 解釈文(JSON) |
| `reports` | 出力レポート | id, result_id, 形式(pdf/png), ファイルパス, 作成日時 |
| `audit_logs` | 監査ログ | id, user_id, 操作, 対象, 日時 |

- 設定・結果・サマリーは可変構造のため **JSON 型カラム**で保持し、手法追加時のスキーマ変更を最小化する。
- アップロードデータ本体はファイル（`uploads` ボリューム）に置き、DB はメタのみ保持する。

## 5. 非同期ジョブ処理フロー

```
[Frontend]            [api/FastAPI]         [Redis]        [worker/Celery]      [DB]
    │  POST /jobs          │                   │                 │              │
    │─────────────────────▶│ ジョブ登録         │                 │              │
    │                      │──────────────────────────────────────────────────▶│ status=queued
    │                      │ enqueue(task)      │                 │              │
    │                      │──────────────────▶│                 │              │
    │  202 + job_id        │                   │ ──pull task────▶│              │
    │◀─────────────────────│                   │                 │ status=running
    │                      │                   │                 │─────────────▶│
    │  GET /jobs/{id}      │                   │                 │ 解析実行      │
    │─(polling)───────────▶│  read status ─────────────────────────────────────▶│
    │  {status:running}    │                   │                 │ 結果保存      │
    │◀─────────────────────│                   │                 │─────────────▶│ status=done
    │  GET /jobs/{id}      │                   │                 │              │
    │─────────────────────▶│                   │                 │              │
    │  {status:done,       │                   │                 │              │
    │   result_id}         │                   │                 │              │
    │◀─────────────────────│                   │                 │              │
    │  GET /results/{id}   │                   │                 │              │
    │─────────────────────▶│ 結果＋解釈返却     │                 │              │
```

- ジョブ状態：`queued → running → done`（失敗時 `error`、メッセージを保持）
- 小サイズ・軽量手法は同期実行にフォールバック可能（その場合も結果は同形式で保存）
- 進捗はポーリング方式（MVP）。将来 WebSocket/SSE 化を検討。

## 6. API エンドポイント一覧（概要）

REST。認証は MVP ではセッション or JWT（詳細設計で確定）。詳細な I/O スキーマは詳細設計書で定義。

| 分類 | メソッド・パス | 概要 |
|---|---|---|
| 認証 | POST `/auth/login` | ログイン |
| 認証 | POST `/auth/logout` | ログアウト |
| ユーザー | GET/POST/DELETE `/users` | ユーザー管理（管理者） |
| データ | POST `/datasets` | CSV/Excel アップロード |
| データ | GET `/datasets` | 一覧 |
| データ | GET `/datasets/{id}` | メタ・プロファイル取得 |
| データ | DELETE `/datasets/{id}` | 削除 |
| 前処理 | GET `/datasets/{id}/profile` | 自動プロファイリング結果（推奨値含む） |
| 前処理 | POST `/datasets/{id}/preprocess` | 前処理設定の適用 |
| 分析 | GET `/methods` | 利用可能手法と設定スキーマ一覧 |
| 分析 | POST `/jobs` | 解析ジョブ登録（手法・設定・dataset_id） |
| 分析 | GET `/jobs/{id}` | ジョブ状態取得（ポーリング） |
| 結果 | GET `/results/{id}` | 指標・グラフ参照・解釈文 |
| レポート | POST `/results/{id}/report` | PDF/画像生成 |
| レポート | GET `/reports/{id}` | レポート取得 |

## 7. セキュリティ（基本設計レベル）

| 項目 | 方針 |
|---|---|
| 通信 | Nginx で HTTPS 終端 |
| 認証 | ログイン必須。パスワードはハッシュ保管（bcrypt 等） |
| 認可 | ロールは admin のみ（MVP） |
| 入力検証 | アップロードファイルの形式・サイズ・内容バリデーション |
| 監査 | 主要操作を `audit_logs` に記録 |
| 保管 | データ本体はホスト内ボリューム（オンプレ完結、外部送信なし） |

※ テナント分離・データ削除依頼フローはクラウド化フェーズで設計（要件定義 10章 参照）。

## 8. 詳細設計への引き継ぎ事項

詳細設計書で確定する項目。

1. 分析手法基底クラスの正確なシグネチャ・共通結果スキーマ（JSON 構造）
2. 各手法の入出力仕様（指標項目・グラフ種別・解釈テンプレート文）
3. DB の物理スキーマ（カラム型・制約・インデックス）
4. API の I/O スキーマ（リクエスト/レスポンス JSON）
5. 認証方式の確定（セッション or JWT）
6. ファイルサイズ上限・想定行数の具体値
7. 前処理ルール閾値の最終確定（欠損率・カーディナリティ等）
8. レポート PDF レイアウト
